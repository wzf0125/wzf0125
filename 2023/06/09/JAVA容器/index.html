构造器

使用空构造器和赋值初始容量构造器区别

空构造器是将elementData指向DEFAULTCAPACITY_EMPTY_ELEMENTDATA 这本质是一个空的Object数组，只有当第一次使用add的时候才会初始化为DEFAULT_CAPACITY(10)

使用初始容量构造器时，若初始容量为0则将elementData指向EMPTY_ELEMENTDATA 标识为空列表

调用add

通过ensureCapacityInternal判断容量是能装下当前数据量+1

通过calculateCapacity判断最少需要的容量

如果是默认容量的空数组，则所需容量为max(10,所需容量),这里不直接返回10是为了兼容addAll方法一次性添加多个元素的情况，可能会添加超过10个元素

然后判断是否需要扩容

如果所需容量大于当前Object数组长度，则需要进行扩容

扩容时机

add/addAll等方法

-> ensureCapacityInternal

-> ensureExplicitCapacity方法

判断所需容量大于当前容量时

扩容大小

1.5倍扩容

旧容量 + 旧容量 >> 1

右移相当于除2

扩容后大小即为1.5倍旧容量

若扩容后大小任不能装下则直接扩容为所需容量(newCapacity < minCapacity)

如果扩容容量大于MAX_ARRAY_SIZE 则使用hugeCapacity进行扩容

MAX_ARRAY_SIZE为 Integer.MAX_VALUE - 8 (-8是因为某些VM胡子爱数字钟保留一些标头字，尝试分配后面8个位置可能会报错内存不足)

序列化&反序列化

ArrayList 的elementData使用了transient修饰 意味着不会被序列化

使用writeObject和readObject进行序列化

  • 序列化过程
    • 读取modCouont(结构被修改次数)
    • 先调用.defaultWriteObject()写入当前类的非静态和非瞬态字段
    • 然后witreInt写入size
    • 再循环写入元素(数组空白部分不会被序列化)
    • 判断写入过程中数组是否被操作过 操作过序列化直接失败
  • 反序列化过程
    • 先初始化elementData为空类型
    • 先调用.defaultReadObject()读取当前类的非静态和非瞬态字段
    • 然后readInt 读取数组大小
    • 通过SharedSecrets.getJavaOISAccess().checkArray(s, Object[].class, capacity);检查反序列化传输数据
      • 判断s是否为Obejct[]类型数据且容量是否大于等于capacity
    • 判断当前数组是否能容纳，不能内部就进行自动扩容
    • 获取当前实例的elementData实例
    • 循环写入数据

扩容全流程

  • 首先调用add / addAll方法
  • 这个时候会做一个容量检查
  • 目标容量是当前元素数量+1 / data.length()(addAll方法的)
  • 首先判断elementData是否是DEFAULT_ELEMENT_DATA是的话说明尚未初始化
    • 指向DEFAULT_ELEMENT_DATA说明只是刚创建但是还没有第一次赋值
      • 虽然说默认大小是10但是如果不进行添加元素的操作的话，初始会指向一个空的Object[] = {} 节省了一定的空间
    • 判断所需容量和默认容量10哪个大，返回大的那个
  • 然后判断所需容量和当前容量情况
    • 若不足则进行自动扩容
    • 计算目标扩容大小
      • 旧容量 + 旧容量 >> 1
    • 若目标扩容大小仍不足则直接扩容至所需大小
    • 若超出MAX_ARRAY_SIZE 则尝试扩容后8位
      • MAX_ARRAY_SIZE => Integer.MAX_VALUE - 8
        • 某些VM会在数组中保留一些标头字，尝试分配更大的阵列可能会导致内存不足错误

HashMap源码分析

map接口基于哈希表的实现，允许所有可选的映射操作，允许null值和null键

初始大小16，每次扩容，容量变为原来的两倍

链表节点

  • hash key 经过扰动得到

  • next 后继

底层存储结构

Node数组 默认大小为16 第一次放入键值时才会初始化为16大小的数组

常量

核心属性

table: 存放核心数据

size: 当前哈希表元素个数

modCount: 当前哈希表结构修改次数(元素个数变动才会增加(修改不会)) (插入，删除元素等)

threshold: 扩容阈值，当你哈希表中的元素超过阈值，触发扩容

loadFactor: 负载因子 一般使用默认0.75

threshlod = capacity(当前数组长度) * loadFactory

构造方法

  • 默认构造方法
  • 自定义大小和负载因子构造方法
    • 参数校验
      • 判断 0 < 容量 < MAXIMUM_CAPACITY (1 << 30)
      • 判断 0 < 负载因子 (这里并没有限制要小于1)
    • 计算扩容阈值tableSizeFor

      • 返回一个大于等于当前cap的一个数字，且这一个数一定是2的次方数

      • 相当于将二进制转换为00011111的形式

      • 比如n = cap (10) -1 = 9

        • 1001

      • 运算后

        • 1111 => 15

      • +1

        • 0001 0000 => 16

链表转换为红黑树时机

当某一链表长度超过8时，且节点总数大于64时进行红黑树转换

桶大小为什么必须是2的次方数

提高性能和拓展性

性能: hashMap的散列方式是以&运算而不是取余运算的

拓展性: 计算索引位置时只需要将hashCode & (length -1)即可

使用2的幂次方作为哈希表的大小还可以方便地进行动态扩容。当哈希表中的元素数量达到一定阈值时，可以创建一个新的哈希表，将所有的元素重新哈希到新的数组中。由于哈希表大小是2的幂次方，因此在重新哈希时可以直接对元素的哈希码进行位运算，而无需进行除法运算，从而提高了性能。

hash方法

散列函数，先判断是否为null值，null值直接hash值为0

否则将key的hashCode异或key的hashCode的高16位(右移16位)

将高位和低位的hashCode做异或操作，目的是为了让hash值更加散列

向hashmap 中put null值时会放在0索引位置

目的: 为了让低16位包含高16位的特征

原因: 由于hashtable 是一个数组 初始未扩容时hashtable的长度length较小

而路由算法是hashCode & (table.length - 1)

例如length只有 16 - 1 => 15

0000 0000 0000 0000 0000 0000 000 1111

此时做&运算就只有后4位

路由算法

就是如何将hashCode散列到散列表中

hashCode & (table.length - 1)

put操作

  • 计算key的hashcode
  • 调用putVal方法
    • 参数:
      • hash (key的hahs值)
      • key
      • value
      • onlyIfAbsent 只有不存在才put(用在putIfAbsent)

变量作用:

tab 当前hashMap的散列表

p: 当前三列表的元素

n: 表示当前散列表的长度

i: 表示路由寻找结果

  • 先判断tab是否为null或散列表长度为0 (延迟初始化逻辑)
    • 进行初始化(扩容)
  • 判断当前所需hash槽是否为空
    • 不存在
      • 则直接在该hash槽放入一个Node
    • 存在则
      • 局部变量: e: node 记录是否存在相同元素 K 临时的key
      • 判断桶头部结合hash值是否相等，且key是否相同( == || equals操作)
        • 相同则将头结点赋值给e
      • 判断头结点是否为树形节点
      • 判断是否是链表节点
        • 遍历链表并记录链表长度
        • 若节点后继为null(迭代到最后一个元素时)新建一个节点连接并结束
          • 判断链表长度是否达到树化阈值
        • 若找到相同节点则break
    • 判断e是否为null(如果是链表不存在该节点的话(插入操作)e会指向null)
      • 不为null说明需要进行节点替换
      • 为null且onlyIfAbsent 为false或旧元素值为null则进行替换操作
        • 替换完成直接返回旧值
  • 判断元素容量是否达到扩容阈值
  • modCount++ (只有插入操作++了，替换操作已经在前面返回了)
  • 达到则调用resize()扩容

核心resize()扩容方法

扩容目的: 为了解决哈希冲突导致的链化严重影响查询效率的问题，哈希表的预期时间复杂度是O(1)，如果链化严重可能会退化为O(n) 所有有必要在合适的时候对哈希表进行扩容和rehash操作

  • 变量解释

    • oldTab： 扩容前的哈希表
    • oldCap：扩容之前那的table数组的长度
    • oldThr：扩容之前的扩容阈值(触发本次扩容的阈值)
    • newCap：扩容之后table数组的大小
    • newThr：扩容之后下次再触发扩容的条件

  • 扩容流程

    • 判断Table是否为null (未初始化的情况) 是则oldCap = 0 不是则等于当前table.length

  • 计算newCap和newThr的值

    • 判断oldCap是否大于0
      • 是则说明散列表已初始化过，这是一次正常的扩容操作
        • 判断是否大于最大大小(2^30)(不能再次扩容)
          • 是
            • 设置下次扩容条件为int最大值(使元素数量无法达到)
            • 返回当前表
          • 否 (新容量小于2^30且大于等于16(初始化赋值大小为2 4 8的情况))
            • 新容量为旧容量2倍(旧容量左移1位)
            • 新扩容阈值为旧扩容阈值2倍(旧)
      • =0则
        • 若下次扩容大小已赋值(调用非空构造方法)
        • 否则赋默认值
        • 容量16 扩容大小12
    • 判断newThr(扩容大小)是否为0
      • 是则说明初始化了小于16的桶，或是新容量大于最大容量的情况
      • 计算newCap * loadFactor 并判断是否超过最大2^30 是的话赋int最大值
    • 赋值threshold为计算出来的下次扩容阈值

  • 创建新的数组[newCap]

    • 判断旧数组为null(未初始化)
      • return newTable
    • 不为null rehash操作
      • 遍历旧桶位，只处理非空头结点
        • 判断头结点是否为单独节点，是则直接rehash到新桶位
        • 为树节点则调用树拆分方法
          • 拆分为高位树和低位树
        • 为链表节点则进行链表拆分
          • 由于hash表为2倍扩容，例如16 => 32
            • 此时15和31索引后四位相同，可以计算链表元素的hash将他们散列到低位和高位桶中
            • 通过当前key & oldCap == 0 ? 低位链表:高位链表 划分高低位链表
          • 判断低位链表是否为null
            • 不为空则将低位链表放到桶中
          • 判断高位链表是否为null
            • 不为空则将高位链表放到桶中
